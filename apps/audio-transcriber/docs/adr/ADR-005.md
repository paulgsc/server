
# ADR-005: GPU Context Pre-Allocation (RTX 4060)

**Status:** Accepted  
**Milestone:** M5  
**Date:** 2025

---

## Context

The current `process_transcription_job` calls `whisper_ctx.create_state()` inside the
worker loop — once per job. On CPU this is fast (a few microseconds). On GPU,
`create_state` triggers `cudaMalloc` which acquires a global CUDA driver lock and
synchronizes with any in-flight GPU operations. The cost is unpredictable and can
reach tens of milliseconds on a loaded GPU.

The RTX 4060 has 8GB VRAM. The Whisper base model requires approximately 150MB for
weights. A full working state (KV cache, mel spectrogram buffers, attention scratchpad)
requires another ~200–400MB depending on context length. This is well within budget.
There is no reason to allocate and free this state on every job.

---

## Decision

Initialize the Whisper GPU context and all working state buffers exactly once during
the plumbing startup phase. Pass a pre-initialized, ready-to-use context to the hotpath
worker. The worker calls the model with the existing context — no allocation, no
initialization, no CUDA sync in the loop.

**Startup sequence:**
1. Load model weights from disk → GPU VRAM (one-time transfer, blocking, in main)
2. Create CUDA stream handles (N streams for pipelining, N fixed at compile time)
3. Allocate all working buffers (KV cache, spectrogram, output tokens) in VRAM
4. Wrap in `GpuWhisperContext` struct and pass to the hotpath via the SPSC

**`GpuWhisperContext` contract:**
- `new(model_path, stream_count)` → allocates everything, blocks until ready
- `run_inference(&self, audio: &[f32; SAMPLES]) -> InferenceResult` → non-allocating,
  reuses pre-allocated buffers, returns stack-allocated or pre-allocated result
- `Drop` → frees VRAM (only called at process shutdown)

**CUDA streams:** Pre-create `STREAM_COUNT` CUDA streams at init. Each stream handles
one inference pipeline stage independently. For the current single-worker architecture,
`STREAM_COUNT = 2` is sufficient (one for H2D transfer, one for compute). This allows
overlapping the next job's host-to-device copy with the current job's compute.

**Pinned host memory:** The audio buffer that feeds H2D transfer must be in page-locked
(pinned) memory to allow `cudaMemcpyAsync`. Allocate one pinned host buffer at init
(`cudaMallocHost`), sized to `SAMPLES * sizeof(f32)`. The real-time audio buffer
`RealTimeAudioBuffer` drains into this pinned buffer — the drain is a `memcpy` from
regular RAM to pinned RAM, which is fine since it happens in the plumbing layer, not
the hotpath.

**VRAM budget (RTX 4060, 8GB):**
```
Whisper base weights:          ~150 MB
Working state (single stream):  ~350 MB
Pinned host audio buffer:        ~0.2 MB (48000 samples × 4 bytes)
CUDA stream overhead:            ~10 MB
OS/driver reserved:             ~500 MB
────────────────────────────────────────
Total:                        ~1010 MB  (well within 8192 MB)
```

If moving to `whisper-large-v3`, the weight budget rises to ~3GB. Still within limits.

---

## Alternatives Rejected

**Alternative A: Pool of pre-allocated states (`Vec<WhisperState>`)**  
Would allow concurrent multi-stream inference on one GPU. Deferred to M6+ because
the current architecture is single-worker. Premature pooling adds complexity without
benefit until multi-stream (ADR-006 scope).

**Alternative B: Re-use existing `whisper_ctx.create_state()` but cache the result**  
This would mean holding a `WhisperState` across jobs in the worker thread. This is
actually what the implementation will do — the distinction from the current code is
that `create_state()` is called once at init, not once per job. The ADR is describing
*where* the call happens (startup, not loop), not whether state is reused.

**Alternative C: TensorRT instead of whisper.cpp + CUDA**  
TensorRT can achieve RTF ~0.05 vs whisper.cpp CUDA's ~0.10–0.15. The tradeoff is:
- TensorRT requires a separate engine compilation step per GPU model
- The compilation step (trtexec) takes 10–30 minutes and is hardware-specific
- The Rust FFI story for TensorRT is immature
Deferred. Start with whisper.cpp CUDA, measure RTF in production, revisit if RTF
target is not met.

---

## Consequences

- `whisper_ctx.create_state()` is completely removed from `worker/whisper.rs`.
  The build script guard (`milestone_m5_no_create_state_in_loop`) enforces this.
- `src/hotpath/gpu.rs` owns the GPU context lifecycle. Only plumbing calls `new()`.
  The hotpath calls `run_inference()`.
- GPU initialization failure is now a startup failure, not a per-job failure. This
  is correct — if the GPU isn't available, the service cannot function. The existing
  NATS retry logic (`connect_with_retry`) is a good model for a `gpu_init_with_retry`
  wrapper.
- The `TranscriptionJob` type changes: it no longer needs to carry `audio: Vec<f32>`
  (heap-allocated). It carries a reference or index into the pre-allocated pinned
  buffer instead. This is a breaking change to the job type — coordinate with the
  SPSC design (ADR-003).
