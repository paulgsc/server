
# ADR-004: Stack-Only k-NN Query Engine

**Status:** Accepted  
**Milestone:** M4  
**Date:** 2025

---

## Context

Speaker diarization and context matching require nearest-neighbor lookups on incoming
audio frame embeddings. A naive implementation would collect results into a `Vec<Neighbor>`,
sort it, and return the top K. This has two problems in the hotpath:

1. `Vec` allocation on every query — unacceptable for a zero-allocation thread.
2. `Vec::sort` uses `pdqsort` which has good average complexity but its worst case
   involves allocations for the scratch buffer when the array is large.

The embedding dimensionality (`DIM`) and the number of indexed entries (`ENTRIES`) are
both known at compile time for a given deployment. K (number of neighbors to return)
is also fixed per query type. All three can be const generics.

---

## Decision

Implement `FixedKnnEngine<const DIM: usize, const ENTRIES: usize>` with a flat centroid
array and `FixedNeighborSet<const K: usize>` as the stack-allocated return type.

**Search algorithm:** Linear scan over all `ENTRIES` centroids. This is O(N) with a
fixed, compile-time N. No tree structure, no dynamic backtracking, no recursion. The
iteration count is exactly `ENTRIES` — deterministic worst case.

**Distance metric:** Squared Euclidean distance (no `sqrt` — monotonic, so ranking is
preserved). The inner loop is `DIM` iterations of `diff * diff + acc`. This compiles
to a vectorized FMA loop on x86-64 with AVX2.

**Result collection:** `FixedNeighborSet<K>` holds `[Option<Neighbor>; K]` and a count.
During the linear scan, each candidate is compared against the current worst neighbor
in the set. If closer, it replaces it. This is an insertion into a bounded sorted array
— effectively an unrolled insertion sort over K elements, where K is typically 1–5.
No heap allocation, no dynamic sort.

```
Worst case instruction count:
  ENTRIES × (DIM FMAs + K comparisons + 1 conditional write)
  = deterministic, compile-time constant
```

**Centroid storage:** `[[f32; DIM]; ENTRIES]` — a 2D array. Row-major layout means
each centroid's DIM floats are contiguous in memory, which is optimal for the inner
loop (sequential cache access).

---

## Alternatives Rejected

**Alternative A: KD-tree (`kiddo` crate)**  
KD-tree average case is O(log N) vs O(N) for linear scan. Rejected because:
- KD-tree backtracking is unbounded in the worst case (degenerate data distributions
  produce O(N) backtracking anyway).
- `kiddo` returns results as a `Vec` or `BinaryHeap` — heap allocations.
- For small ENTRIES (< 1000), linear scan with AVX2 vectorization outperforms
  KD-tree due to cache effects.

**Alternative B: HNSW (`hnswlib` via FFI)**  
HNSW is the right answer for large-scale ANN at inference time. Not appropriate here:
- FFI cost per query adds latency on the hotpath.
- HNSW graph is dynamically allocated.
- The entry count for speaker embeddings in a single stream session is small (< 100).

**Alternative C: Runtime-sized `Vec<Neighbor>` with capacity pre-set**  
Same objection as ADR-002 Alternative A. Pre-allocated Vec still risks realloc if the
caller exceeds K results. Fixed array makes the bound a compile-time guarantee.

---

## Consequences

- `DIM` and `ENTRIES` become compile-time constants. Changing them requires recompile.
  For a service binary this is fine. If dynamic embedding dimensionality is needed
  in the future, this decision must be revisited.
- The `FixedNeighborSet<K>` type is `Copy` (all fields are fixed-size and Copy),
  which means it can be returned from functions without move semantics overhead.
- The `FixedKnnEngine` centroid data is read-only after initialization. It can be
  placed in a `static` or initialized once at startup. It does not need to be behind
  a mutex or Arc.
- The `insert_if_closer` function inside the search loop must be `#[inline(always)]`
  to ensure the compiler unrolls the K-element insertion loop.
