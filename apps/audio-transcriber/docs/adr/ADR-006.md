## Decision (Logging isolation)

`info!`, `warn!`, `debug!` macros from `tracing` must be removed from the hotpath
execution loop. The `tracing` subscriber internally uses a `Mutex` to serialize writes.
Calling it from a real-time thread introduces unbounded latency.

**Chosen approach:** Atomic ring-buffer appender. The hotpath thread writes log records
into a pre-allocated `[LogRecord; LOG_CAP]` ring buffer using atomic index advancement
(same pattern as the SPSC in ADR-003). A separate plumbing-side thread drains this
buffer and forwards records to the `tracing` subscriber.

**Format of `LogRecord`:**
```rust
pub struct LogRecord {
    pub level: u8,          // 0=trace 1=debug 2=info 3=warn 4=error
    pub timestamp_ns: u64,  // from CLOCK_MONOTONIC, no syscall on Linux vDSO
    pub msg: [u8; 256],     // fixed-size message buffer, null-terminated
    pub len: u8,
}
```

`LOG_CAP` = 64 records. If the drain thread falls behind, oldest records are silently
overwritten — same policy as audio drops. This is observable via a `logs_dropped`
counter.

## Decision (End-to-end `TranscriptionJob` type)

With the SPSC replacing mpsc and the GPU context pre-allocated, `TranscriptionJob`
no longer needs a `Vec<f32>`. The revised type:

```rust
pub struct TranscriptionJob {
    pub seq: u64,
    pub audio_frame_idx: usize,  // index into pre-allocated pinned buffer
    pub sample_count: usize,
    pub sample_rate: u32,
    pub created_at: u64,         // nanoseconds, from atomic clock
}
```

The audio data itself lives in the pre-allocated pinned host buffer (ADR-005). The
SPSC carries only this small descriptor struct. This keeps the SPSC slot size small
(fits in a cache line) and avoids any copy of audio data across the boundary.
