
# ADR-002: Zero-Allocation Fixed-Size Audio Buffer

**Status:** Accepted  
**Milestone:** M2  
**Date:** 2025

---

## Context

The existing audio processor accumulates samples into a `Vec<f32>` that grows
dynamically. In a real-time audio callback context this is fatal: `Vec::push` can
trigger a `realloc` at an unpredictable moment, stalling the thread for an
indeterminate duration while the allocator runs. On a loaded system this can be
milliseconds — enough to cause an audio frame underflow.

The real-time buffer's job is simple: hold up to `CAP` samples of f32 audio,
expose a `push_frame` that is O(1) with a fixed execution bound, and expose a
`drain` that hands off the filled buffer to the SPSC.

---

## Decision

Replace the `Vec<f32>` audio buffer with `RealTimeAudioBuffer<const CAP: usize>`.

```rust
pub struct RealTimeAudioBuffer<const CAP: usize> {
    samples: [f32; CAP],
    write_ptr: usize,
}
```

**Sizing:** `CAP` is derived from `target_sample_rate * buffer_duration_secs` and
is set as a compile-time constant. The existing config values (16000 Hz, 3s default)
give `CAP = 48_000`. This is a stack allocation of 192KB — well within the default
thread stack size (8MB on Linux). If needed for larger buffers, the struct can be
placed on the heap once at init time via `Box::new(RealTimeAudioBuffer::default())`,
but the allocation happens at startup, not in the hotpath.

**Overflow behavior:** `push_frame` returns `Err("overflow")` when `write_ptr + frame.len() > CAP`.
It does NOT grow, does NOT panic, does NOT wrap. The caller (SPSC producer) is
responsible for draining the buffer before overflow occurs. Overflow is a logic
error indicating the SPSC consumer is too slow — it should be counted as a metric
and the frame dropped, consistent with the existing backpressure policy.

**Drain semantics:** `drain` copies `samples[0..write_ptr]` into a fixed-size output
array (also stack-allocated), then resets `write_ptr = 0`. The copy is a single
`copy_from_slice` call — one `memcpy` with a bounded size.

---

## Alternatives Rejected

**Alternative A: Keep `Vec<f32>` with pre-allocation (`Vec::with_capacity`)**  
Rejected. `with_capacity` prevents realloc on push, but does not prevent realloc
if `push` exceeds capacity (it grows silently). The developer must remember to
never exceed capacity. A fixed array makes exceeding capacity a compile-time or
explicit-error condition.

**Alternative B: `arrayvec::ArrayVec<f32, CAP>`**  
Reasonable alternative. `arrayvec` is well-maintained and provides the same
guarantee. Rejected in favor of the hand-rolled version to avoid a new dependency
and to keep the implementation auditable — it's 40 lines of code, not a crate.
If this decision is revisited, `arrayvec` would be the first choice.

**Alternative C: Ring buffer for the audio accumulation buffer itself**  
Rejected. A ring buffer is the right structure for the SPSC (see ADR-003), but
not for sample accumulation. Sample accumulation is append-only until the buffer
is full, then fully drained. This is exactly the sequential write pattern that
a flat array with a write pointer handles optimally.

---

## Consequences

- `buffer_duration_secs` is no longer a runtime-configurable value that affects
  allocations — it is baked into the type at compile time. If operators need to
  change it, they recompile. This is acceptable for a service binary.
- The existing `AudioProcessor.buffer: Vec<f32>` field is replaced. The `audio.rs`
  module becomes a plumbing adapter that feeds the new buffer type.
- The custom allocator test harness (`tests/alloc_guard.rs`) wraps push_frame and
  drain calls and asserts zero allocations. This is the definitive verification.
