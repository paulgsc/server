
# ADR-003: SPSC Ring Buffer at the Hotpath Boundary

**Status:** Accepted  
**Milestone:** M3  
**Date:** 2025

---

## Context

The current hotpath boundary uses `tokio::sync::mpsc::Sender<TranscriptionJob>` with
`try_send`. This works correctly for backpressure, but has two problems for the
GPU transition:

1. **Multi-producer overhead.** `tokio::sync::mpsc` supports multiple producers and
   pays for that with internal locking on the sender side. In the new architecture
   there is exactly one producer (the async audio ingestion loop) and exactly one
   consumer (the GPU worker thread). The MPSC generality is wasted cost.

2. **Async executor coupling.** Even `try_send` on a Tokio channel acquires an internal
   lock on some paths (notification of the receiver's waker). The GPU worker thread
   is a real-time OS thread — coupling it to Tokio's scheduler via a channel that
   notifies Tokio wakers is architecturally incorrect. The consumer side (`blocking_recv`)
   parks on a Tokio primitive, which can introduce scheduler jitter.

The replacement must be: single-producer, single-consumer, lock-free, non-blocking on
both sides, statically bounded in capacity.

---

## Decision

Implement `SpscRingBuffer<T, const N: usize>` with two `AtomicUsize` indices.

```
Producer side (async plumbing):  try_push  — Relaxed load of tail, Acquire load of head,
                                             Release store of new tail
Consumer side (RT worker thread): try_pop  — Acquire load of tail, Relaxed load of head,
                                             Release store of new head
```

**Memory ordering rationale:**

- `tail` is written by producer, read by consumer → producer writes with `Release`,
  consumer reads with `Acquire`. This establishes a happens-before: the data write
  into `buffer[tail]` happens before the consumer reads it.

- `head` is written by consumer, read by producer → consumer writes with `Release`,
  producer reads with `Acquire`. This establishes a happens-before: the slot is
  available for reuse before the producer writes to it.

- No `SeqCst` is needed. The acquire-release pairs are sufficient for a two-party
  protocol. `SeqCst` adds a full memory fence that is unnecessary here and costs
  performance on aarch64.

**Capacity:** `N` must be a power of two to allow index masking instead of modulo.
`(index + 1) & (N - 1)` vs `(index + 1) % N` — the former is branchless and compiles
to a single AND instruction.

**Slot storage:** `[MaybeUninit<T>; N]` rather than `[Option<T>; N]`. `Option<T>`
adds a discriminant byte (or more, depending on T's niche optimization). With
`MaybeUninit`, the producer writes directly into the slot and the consumer reads
directly out. The indices are the source of truth for validity — not a discriminant.

**Ownership:** The buffer is wrapped in `UnsafeCell` to allow shared references from
both producer and consumer. Safety is upheld by the atomic index protocol above.
The struct is `!Sync` for `T: !Send` and explicitly marked `Send` only when `T: Send`.

---

## Alternatives Rejected

**Alternative A: `crossbeam::channel::bounded`**  
A correct and well-audited choice. Rejected because it is MPMC (multi-producer,
multi-consumer) and pays for generality we don't need. Also adds a dependency.
The hand-rolled SPSC is ~80 lines and is fully auditable against the spec in this ADR.

**Alternative B: `rtrb` crate**  
`rtrb` is specifically designed for real-time audio SPSC. Seriously considered.
Deferred in favor of the hand-rolled version for the same auditability reason as
`arrayvec` above. If the hand-rolled version shows subtle bugs under the threaded
test (M3: `milestone_m3_single_producer_single_consumer`), switch to `rtrb`.

**Alternative C: Keep `tokio::sync::mpsc::try_send`**  
Rejected. See context. The architecture doc is explicit: the hotpath must not depend
on the async executor. `try_send` on a Tokio channel is not executor-free.

---

## Consequences

- `process_audio_chunk` in `main.rs` becomes the single SPSC producer. It calls
  `try_push` and handles `Err` (full) the same way it currently handles
  `TrySendError::Full` — increment drop counter, log warn.
- The GPU worker loop becomes the single SPSC consumer. It calls `try_pop` in a
  tight spin loop with a short sleep on `None` to avoid 100% CPU burn while idle.
  This replaces `blocking_recv`.
- The `TranscriptionQueue` wrapper struct in `worker/queue.rs` is replaced by the
  SPSC directly. The sequence counter moves into the producer side.
- Tests must verify the threaded case (one thread pushing, one thread popping
  concurrently) under Miri or with a stress loop to catch any ordering bug.
