
# ADR-001: Hotpath Module Isolation

**Status:** Accepted  
**Milestone:** M1  
**Date:** 2025

---

## Context

The existing `audio-transcriber` binary mixes async I/O concerns (NATS ingestion,
observability, lifecycle management) with CPU-bound compute concerns (Whisper
transcription, audio buffering) in the same module tree. The proposed GPU transition
requires strict separation because:

1. Real-time thread safety guarantees are incompatible with async executor assumptions.
   Tokio can park a task on any OS thread — a real-time audio callback must never
   be scheduled this way.

2. Zero-allocation requirements cannot be audited when async code and hotpath code
   share modules. The allocator guard tests (M2) would be meaningless without a
   clear boundary to test against.

3. Clippy lint rules (`pedantic`, `nursery`) that enforce the hotpath constraints
   would generate excessive false positives on async plumbing code if applied globally.

---

## Decision

Create a dedicated `src/hotpath/` module subtree with the following invariants,
enforced by the combination of lint configuration, forbidden imports, and tests:

**Forbidden inside `hotpath/`:**
- `tokio::*` (any async runtime dependency)
- `std::sync::Mutex` or `std::sync::RwLock` (blocking primitives)
- `async fn` (async function definitions)
- `.await` (async call sites)
- Dynamic allocation via `Vec::new()`, `String::new()`, `Box::new()`, etc., except
  in explicit `_init` functions that run once at startup

**Allowed inside `hotpath/`:**
- `std::sync::atomic::*` (non-blocking synchronization)
- `const` generics for all buffer and collection sizing
- FFI calls into CUDA/whisper-rs (these are blocking by nature and belong here)
- `#[inline(always)]` on the critical path

**Module structure:**
```
src/hotpath/
  mod.rs       — public API surface, re-exports, module-level lint attrs
  buffer.rs    — RealTimeAudioBuffer<CAP>
  spsc.rs      — SpscRingBuffer<T, N>
  knn.rs       — FixedKnnEngine + FixedNeighborSet
  gpu.rs       — GPU context wrapper (pre-allocated)
  README.md    — invariant contract (human-readable)
```

The boundary between plumbing and hotpath is the `SpscRingBuffer`. Plumbing pushes
raw audio chunks into it. The hotpath thread pops from it. Nothing crosses this
boundary except `T` (a fixed-size audio frame type).

---

## Alternatives Rejected

**Alternative A: Keep everything in one module, rely on code review**  
Rejected. Code review doesn't catch accidental `Vec` growth in a fast path six months
later when you've forgotten the constraint. The compiler and tests do.

**Alternative B: Separate crate (`hotpath-rt`)**  
Considered. A separate crate would enforce the boundary even more strongly (no
accidental internal imports). Deferred because the overhead of a new workspace member
isn't justified yet — the module-level lint attributes and forbidden-import tests
achieve the same audit capability. Revisit at M6 if integration is messy.

**Alternative C: `#![no_std]` on the hotpath module**  
Blocked by whisper-rs and CUDA FFI dependencies, which require `std`. The custom
allocator guard in tests achieves the same zero-allocation verification at test time.

---

## Consequences

- The `src/hotpath/mod.rs` will carry `#![allow(clippy::...)]` suppression attributes
  for a small set of rules that conflict with real-time patterns (e.g., the rule
  against raw pointer arithmetic in the SPSC implementation).
- All other modules must never import from `src/hotpath/` except through the public
  re-exports in `hotpath/mod.rs`. This keeps the API surface small and auditable.
- Adding anything new to `hotpath/` requires a corresponding test in
  `tests/milestones.rs` and a reviewer to confirm the invariants hold.
