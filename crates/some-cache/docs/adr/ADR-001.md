# ADR-001: Replace `moka` + `fastrand` with a hand-rolled single-flight cache

**Status:** Accepted
**Milestone:** M2 ("mechanical sympathy")
**Date:** 2026

---

## Context

`some-cache` is the shared Redis caching contract for the axum server and the
pipeline daemon. On top of the Redis round-trip, `DedupCache` (`src/dedup.rs`)
adds an in-process **thundering-herd guard**: when many requests miss on the
same key at once, only one runs the expensive upstream fetch; the rest coalesce
onto its result. That guard is currently implemented with
`moka::future::Cache<String, Arc<[u8]>>`.

We use exactly three things from moka:

1. `try_get_with(key, init)` — single-flight: run `init` once per key, others await it.
2. `max_capacity(n)` — bound the number of in-flight/cached entries.
3. `remove(key)` / `invalidate_all()` — invalidation.

That is a sliver of moka's API. In return, moka pulls a transitive tree to
deliver a concurrent W-TinyLFU cache we are barely exercising. A `cargo tree`
on this crate shows the cost — `crossbeam-epoch`, `crossbeam-channel`,
`tagptr`, `async-lock`, `event-listener`, `quanta`, `arc-swap` — none of which
appear anywhere else in our dependency graph for this purpose.

Separately, `src/store.rs::should_touch` depends on the entire `fastrand` crate
for a single biased coin flip (`fastrand::f64() < p`).

Two forces motivate this ADR:

- **Dependency hygiene** ("less pip-install-the-universe"). Every crate we pull
  is supply-chain surface, compile time, and binary size. A guard this small
  does not justify a concurrent-cache framework.
- **Mechanical sympathy as a learning vehicle.** The replacement is small enough
  to build by hand, but rich enough to force real mastery of sharding, false
  sharing, cache-friendly eviction, and cancellation-safe async coordination.
  This milestone is run as a tutored exercise (see `docs/USER_STORY.md`), not a
  code-generation task — the value is the understanding, and the artifact is the
  proof.

---

## Decision

Build `src/inproc/` — a sharded, cache-line-aware, single-flight in-process
cache — from `std` plus the `tokio` we already depend on, and delete `moka` and
`fastrand`. The module is delivered in milestones, each gated by tests:

```
src/inproc/
  mod.rs            — module lint contract + `InProcStore` trait (the minimal moka surface)
  rng.rs            — thread-local probability gate            [M2.0 — replaces fastrand]
  cache.rs          — `InProcCache`: shard array + single-flight + try_get_with [M2.1/M2.5]
  shard.rs          — `#[repr(align(64))]` lock-padded shard   [M2.2 — false sharing]
  lru.rs            — CLOCK bounded eviction                    [M2.3]
  single_flight.rs  — leader/follower coalescing + cancellation [M2.4]
```

**Invariants, enforced by the combination of a module lint contract and tests:**

- `inproc` carries a scoped `#![deny(...)]` block (`clone_on_ref_ptr`,
  `rc_buffer`, `mutex_atomic`, `trivially_copy_pass_by_ref`, `mem_forget`,
  `unsafe_op_in_unsafe_fn`). These fail the build if the implementation drifts
  back toward allocation-happy, refcount-churning patterns. They are scoped to
  this subtree because the rest of the crate is async plumbing where some would
  be noise; promote them crate-wide only after cutover (M2.5).
- Steady-state `get`/`insert` and the probability gate make **zero heap
  allocations**, proven under the `tests/alloc_guard.rs` guarded allocator.
- `Shard` is ≥64-byte aligned so adjacent shard locks never false-share.
- Capacity is a hard bound: a full cache stays exactly at capacity under churn.
- Single-flight runs `init` exactly once under N concurrent callers, and is
  cancellation-safe: cancelling the leader must not hang the followers.

`tests/milestones.rs` is the definitive source of truth for progress. A
milestone is done when its tests are green AND its `#[ignore]` markers are gone
AND the lint contract holds.

---

## Alternatives Rejected

**Alternative A: Keep moka.**
Rejected. It is the status quo this ADR exists to change. It works, but it is a
large dependency for a tiny need and teaches nothing. The point of M2 is both
the leaner tree and the mastery built getting there.

**Alternative B: Swap moka for a smaller off-the-shelf crate (`quick_cache`, `lru`, `mini-moka`).**
Rejected for this milestone. It would trim the tree but still be "pip-install
the answer." It also wouldn't deliver the learning objective — the whole reason
to do this by hand is to internalize sharding, false sharing, CLOCK, and
cancellation safety. (If the hand-rolled version ever proves inadequate under
real load, revisiting a vetted crate is a legitimate future ADR.)

**Alternative C: Drop the in-process guard; rely on Redis alone.**
Rejected. Redis cannot see upstream fetch latency or coalesce concurrent misses
before they hit the upstream. `FETCH_DURATION` and `DEDUP_WAITERS` (see
`src/metrics.rs`) exist precisely because this guard absorbs herd pressure Redis
is blind to. Removing it changes behavior; this ADR is a like-for-like swap.

**Alternative D: A single global `Mutex<LruMap>` instead of sharding.**
Rejected. One lock serializes every cache op across all cores — the contention
moka shards away. We keep the sharding; we just own it.

## Consequences

- `src/inproc/mod.rs` carries a temporary `#![allow(dead_code)]` while bodies are
  `todo!()`. Removing that allow is part of the M2.5 definition of done.
- M2.4 will likely enable tokio's `sync` feature (for `Notify`) and add tokio
  `rt`/`macros` as **dev-dependencies** for the concurrency tests. Adding a
  feature/dev-dep to a crate we already use is in-scope; adding a new
  third-party crate is not.
- After cutover, `cargo tree -p some-cache` should no longer list the moka
  subtree, and the `multiple_crate_versions` clippy noise sourced from it should
  shrink. This is observable and is part of the acceptance evidence.
- Until M2.5, `inproc` is dead code compiled alongside the live moka path; the
  crate keeps building and shipping normally throughout the sprint.
- This ADR deliberately scopes out the HTML/regex parsers and other crates.
  M2 is one crate, one axis. (Mirrors how `audio-transcriber/docs/adr/ADR-001`
  deferred unsafe until the foundations were in place.)
